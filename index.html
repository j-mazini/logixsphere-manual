Login & Initial Functions

The login screen was the first module structured in the system. It guarantees that only authorized users reach the platform, protecting operational, financial and registration data while keeping every user within their permission level.

Main screen functions

  • Enter e-mail & password to sign in
  • Recover credentials via Forgot Password
  • Run the First Login flow on initial access
  • Auto-routing to areas allowed by profile

The first Administrator

When the system has no registered users yet, the first Admin profile is created exclusively by the development team. That Admin then gains full autonomy to run the administrative structure.

Dev-created

Administrator autonomy

  • Register new users & other Admins
  • Register Vendors
  • Configure access levels
  • Manage operational, financial & admin modules

First Login flow

Designed so a Vendor or pre-registered user can securely create their own password.

The Administrator pre-registers the user or Vendor.

The user opens the system's initial screen.

They select the First Login option.

The system requests the previously registered e-mail.

After validation, the user is directed to create a password.

Access is released according to the assigned profile & permissions.

01·5

Roles & the Access Matrix

Logixsphere has three user types, identified by the user_type_id carried inside the JWT token. Everything a person can see or do flows from that single number. This is the map of who reaches what — page by page.

Three user profiles

One field in the token decides the entire experience. There are no sub-roles — just these three.

Profileuser_type_idIn one sentence
Admin1System owner: master data + planning + execution + finance + users. No fences.
Driver / Vendor2The field provider: sees their own performance, their own invoices and their own requests.
Supervisor3Operations manager: plans and executes like the Admin, but with no finance and no master data.

Control happens in three layers

Frontend — screen whitelist (the main barrier). The middleware reads user_type_id on every page request. Admin passes straight through; a Vendor only reaches its allow-list, anything else redirects to /daily-performance-insight; a Supervisor only reaches its allow-list, anything else redirects to /daily-game-plan.

Frontend — sidebar assembly. The side bar hides whole sections per profile: the Supervisor sees neither the Finance section nor master data; the Vendor sees almost nothing beyond its own world.

Backend — route middlewares. Each endpoint demands a level: authenticateAdminJWT (Admin only — 85 endpoints: all master data, all finance, mass deletes, request approval), authenticateAdminOrSupervisorJWT (Admin or Supervisor — 26 endpoints: planning, game plan, operation transitions), and authenticateJWT (any authenticated user — most read GETs).

Security note — the real barrier is the frontend

Most read (GET) routes use only authenticateJWT, so at the API level any logged-in user could read a lot. What actually stops a Vendor from reaching admin screens is the frontend whitelist, not the backend. The true data locks in the backend are pointed and specific (see below).

Where the backend really locks data

Beyond the screen whitelist, four specific guards enforce data ownership at the API:

  • Anti-IDOR

    Deductions. Vendor and Supervisor read only their own — swapping the :userId in the URL returns 403. Only the Admin reads other people's.

  • Scoped

    Service Partner. Vendor and Supervisor are pinned to the SP in their own token; only the Admin picks an arbitrary SP via query.

  • Scoped

    Week Planner exports. Admin and Supervisor export any user's data; the Vendor exports only their own.

  • Scoped

    Chat. The Vendor talks only to Admin / Supervisor; Admin and Supervisor talk to everyone.

Pocket map — what each profile lives in

👤 Vendor · type 2

Lives in its own world. Sees only: daily-performance-insight (landing), subcontractor, deductions (own), current-month, current-performance, mobile-invoice (own) and requests. Any other route → back to daily-performance-insight. In chat, talks only to Admin/Supervisor.

👔 Supervisor · type 3

Operations manager without a wallet. Everything the Vendor has plus home, week-planner (incl. Generate Week), daily-game-plan, daily-operations-reports, vendor-performance and requests-inbox. No finance, no master data, no request approval, and reads only its own deductions. Off-list route → back to daily-game-plan.

🛡️ Admin · type 1

No fences. All official pages and the legacy ⚠️ ones. The only profile that creates/edits master data, runs all finance, manages users, approves requests, mass-deletes, sees anyone's data and crosses the Service Partner boundary. The trade-off: on mass, irreversible and financial actions the Admin is the last check that exists.

02

Master Data & Setup

The configuration backbone of the platform: where service providers, logistics entities and auxiliary records are created and become the master source feeding every other module.

3 · Vendors — the central database

Vendor Management is one of the largest modules by volume of information. It works as the central database of Logixsphere's service providers — concentrating registration data, documents and contractual links used across the platform. Acting as a master data source, vendor records feed:

  • Contract generation
  • Cost-model definition
  • Route association & document control
  • Finance integration & invoice generation

4 · Contract & Routes Management

A central parameterization area where depots, routes, loops, regular services and cost centers are created. It is split into three tabs:

A · Depot Management

Create & organize logistics entities: depots, loops, routes, regular service types and the associations between them — feeding Week Planner & Daily Game Plan.

B · BA Cost Centers

Register internal services/tasks with no external revenue source — a cost center for expenses, payments and internal allocations tied to Vendors.

C · Ad-Hoc Services

Register extra services outside fixed routes. The system auto-classifies them so they are correctly identified in operational and financial modules.

5 · Assets — auxiliary registries

A hub of auxiliary records that keeps the platform organized, standardized and aligned with the real operation.

Vendor Type

Service types provided by Vendors and the associated vehicle/position.

User Type

User types & access levels — controls permissions and sensitive screens.

Vehicle Type

Vehicle types used in service — aids route planning & fleet control.

Team Support

Internal support roles for team organization & responsibilities.

AdHoc Category

Categories later used in invoices, reports & financial classification.

03

Operations

From the central dashboard to weekly planning and live field execution — the modules that turn schedule intent into validated, billable operational data.

2 · Homepage — the central panel

The first screen the Administrator sees after login. A dynamic dashboard of cards and info blocks offering a fast, consolidated view of indicators, pending items, fleet status, documents and financial summaries. Cards are customizable & reorderable. Supervisors see a limited version.

Day View

Current-day operations in real time — active depots, routes, vendors & vehicles.

Operational Summaries

Consolidated prior-day data, fed automatically via DHL's SPMS & OPMS.

Financial Summaries

Consolidated balance of revenue, costs & results from earlier periods.

Liquidation Damage

Deductions & penalties from operational failures against DHL.

Document Compliance

Courses, documents & requirements tied to registered drivers.

Vehicle Compliance

Fleet regularization status — documentation & operating conditions.

6 · Week Planner — the starting point

Presents the week as a seven-day grid (Mon–Sun) so management can organize drivers, vehicles and routes in advance — the base for the Daily Game Plan and downstream finance. It uses a Drag & Drop interface for fast, intuitive adjustments.

Generate Week

Auto-builds the schedule from each driver's default routes; inactive drivers are skipped.

Copy to Next Week

Duplicates the current structure to the following week to cut manual entry.

Day-Off Spare

Any active driver left unassigned automatically gets a Day Off record.

Save / Assign All Week

Apply a change to one day, or replicate driver, vehicle & sort across the week.

Temporal safety lock

Only future dates are editable; current & past dates stay read-only, protecting the financial and operational history. Admins hold global access to all depots through the centralizing BAOP depot.

7 · Daily Game Plan (DGP)

The field operations core. Supervisors start, track, adjust and close the day's real operations. It consumes the Week Planner data and allows real-time changes — absences, substitutions, route changes or emergency demands.

Real-time

DGP — Blocking Flow & Finish

If any day in the last seven is pending, the system blocks the current day until it's resolved — ensuring consistency for audit and finance. Finish batches validated data to accounting audit.

Safety gate

8 · Daily Operations Management (DOM)

The final filter before billing — review, adjust and validate the operational data closed by Supervisors. Fine adjustments cover decimal hours, overtime, contractual rates, driver status, operational inconsistencies and finance preparation.

Error alerts & Bulk Submit

If a driver worked but lacks a proper record, the system creates a synthetic line forcing regularization before data reaches finance. Bulk Submit then marks records as Submitted, locking edits and releasing data for official invoices.

14 · Requests — the field channel

Where any user creates and tracks their own requests (vendor requests) — the way a driver asks for something: a deduction, a correction, a day off, an ad-hoc need. All three profiles (Admin, Supervisor, Vendor) reach this screen; it is the Vendor's single channel to management.

Open to everyone

15 · Requests Inbox — management side

The inbox where incoming requests are managed, with a month selector and a badge counting the current month's pending items. Admin and Supervisor both see the inbox (the sidebar counts pendencies for both); the Vendor has no access.

Approval is Admin-only

Approving, editing or deleting a request is exclusive to the Admin. The Supervisor sees the inbox, but the final decision belongs to the Admin.

Transforming operational data into reports for management, auditing and decision-making — across the whole operation or down to a single driver.

9 · Daily Operations Insights

A consolidated analytical view of logistics performance over a period. It applies advanced hour rules, automatically adding Sort Hours per depot plus a single daily Driver Extra Hours bonus of 15 minutes.

LCY+2h sort
MSE+1h45m sort
All drivers+15m daily bonus

It consolidates total stops, total records, exclusive routes, per-depot & per-driver data and general metrics — exportable to Excel or PDF.

10 · Vendor Performance

A management panel for analyzing service-provider performance over the month. With no driver selected, it shows a consolidated fleet view; an autocomplete search filters to a single driver, and the calendar adjusts to only the months they actually worked. It works on the monthly average of the key KPIs:

Avg SPR

Avg SPOR-H

Adjusted hourly agility.

Avg AFD

Failures, misses & occurrences.

05

The Financial Ecosystem

From daily financial forecasting to deductions and the final invoice closing — the modules that turn validated operations into billing, with strict access rules.

11 · Daily Financial Insights

The first screen of the financial ecosystem. It brings economic predictability — letting administration track revenue, costs and margins before official invoices are generated. A daily, consolidated control panel covering:

  • Gross revenue & accumulated costs
  • Per-depot data & submitted rates
  • Margin auditing, day by day

12 · Deductions — accounting intelligence

Registers charges, penalties, vehicle repairs, salary advances and other debts tied to Vendors (also receiving input from Vendor Requests). Deductions are organized into categories:

Repairs
Traffic Penalties
Pre-Payments
Liquidation Damages

Charge dates are calculated automatically, skipping weekends & holidays via a banking-business-day check. Each entry gets a unique, human-readable serial code based on debt type, employee initials and date.

Traceable serial codes

Example: RPD-JS-020626-01. When a deduction is linked to a closed invoice, its status auto-changes to Paid — ensuring traceability and avoiding duplicate charges.

13 · Invoices — the final block

The last structured block of the financial ecosystem. It gathers operational, financial, contractual and deduction data to close payment invoices for drivers and subcontractors. Batch issuance lets the Administrator close subcontractor accounts in a consolidated way, and supports consolidated billing based on the last 12 months of data.

Strict access security

Drivers only see a statement-style view (My Payslips), mobile-friendly. Supervisors have no access to accounting data and are auto-redirected if they try. The Administrator is the only profile with full financial access — closing, checking, issuing and auditing invoices.

Two deduction screens — don't confuse them

The system has two distinct deduction surfaces. Mixing them up is a common source of confusion:

Operational · /deductions

A consultation view of the deductions applied to a user. Admin sees anyone's and can pick an arbitrary Service Partner; Supervisor and Vendor see only their own (403 on another userId — the anti-IDOR lock).

Financial · /deductions-disbursements-recharges

The Admin-only financial screen that gathers the five deduction types plus disbursements and recharges. Hidden from Supervisor and Vendor entirely.

Cautions & Attention Points

A practical guide to avoiding problems while using the system. Each point is classified by impact level so you know where discipline matters most.

The lifecycle of an operation

Every operation moves through three stages. Once it is finished, it cannot be reopened — this is the single most important rule in the system.

Not Startednothing recorded yet
→
In Progressday being operated
→
Finishedlocked · final
Critical · can stall the operation or affect financial data Attention · disrupts flow or needs discipline Minor · annoyance, no real risk

No going back

Trying to revert a closed day results in an error; the data must then be corrected through alternative paths (deductions, financial adjustments). Respecting each operation's cycle is what sustains the platform's reliability.

Records

  • Attention

    Forgetting a Vendor's route or default vehicle blocks automatic allocation for the week.

  • Critical

    Duplicate plates cause allocation failures — always check whether a plate already exists.

  • Minor

    Special vehicles like "Off" must not be deleted — the system uses them as markers.

Week Planning

  • Critical

    Enter the days off before generating the week. Running auto-generation without them produces a wrong result and forces a redo.

  • Critical

    Generation in large depots can show partial failures — always review the result after generating.

Daily Operation

  • Attention

    Blocking flow: a pending day within the last 7 days stops you from advancing. Check past pendencies before starting the day.

  • Critical

    Batch actions (mass close) cannot be undone — confirm everything before executing.

Auditing & Finance

  • Critical

    Batch submit to finance locks edits; errors after that are hard to revert — always review before submitting.

  • Critical

    Revenue calculation uses complex rules (rate bands). Small payment-model errors can produce incorrect amounts.

  • Attention

    Some invoices may display cached data — refresh the view when in doubt.

Reported symptomLikely cause
"I can't edit this day"The date has already passed, or the day is finalized.
"The driver vanished from the schedule"The record is inactive, or it was marked as "Spare".
"The day won't open"There's a pendency in the last 7 days, or no leader assigned to the depot.
"The invoice came out with the wrong amount"Payment model or rate band registered incorrectly.
"I can't delete a vehicle"It's a system vehicle (e.g., a day-off marker).
